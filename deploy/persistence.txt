PERSISTENCE — what can and cannot be promised
==============================================

No software vendor can *guarantee* that a cloud VPS will "never" shut down. The
host can migrate, patch, or restart nodes; the guest OS can OOM, panic, or lose
power; your account or disk can fill.  What *is* achievable is:

  * Services that **start at boot** (``systemctl enable``)
  * Processes that **restart when they exit** (``Restart=always`` in systemd, or
    Docker ``restart: unless-stopped``)
  * So that **normal** crashes and **reboots** bring the stack back without you
    logging in over SSH

That is the practical maximum. For proof after you configure the server, reboot
the VPS once and confirm everything came back (IB Gateway, your systemd units,
Streamlit, master loop).

Shipped in this repo (adjust paths to match your install, e.g. ``/opt/``)::

  - ``deploy/rlm-control-center.service.example`` — Control Center (Streamlit)
  - ``deploy/linux/regime-locus-master.service`` + ``deploy/linux/install-systemd.sh`` — master loop
  - ``deploy/systemd/ib-gateway.service.example`` — optional: ``docker compose up -d`` before RLM (see ``deploy/ib-gateway-compose.example.yml``)
  - ``deploy/systemd/rlm-preopen*.example`` / ``rlm-postclose*.example`` — optional timed session briefs (``scripts/run_session_brief.py``)

Boot ordering
-------------

Install and **enable** IB Gateway (Docker or ``ib-gateway.service``) before the master loop so API
TCP is accepting connections; ``regime-locus-master.service`` lists ``After=docker.service ib-gateway.service``.
If you do not use a systemd unit named ``ib-gateway.service``, edit ``After=`` in your local copy or
expect a short burst of connection errors until Gateway is healthy.

**Reboot checklist:** ``systemctl is-active docker`` (if used), ``ib-gateway`` (or verify container),
``regime-locus-master``, ``rlm-control-center`` if installed; ``journalctl -u regime-locus-master -b``
for IBKR **502** / auth messages.

IBKR / unattended limits
------------------------

Fully unattended operation is **not** guaranteed: periodic **2FA**, password changes, **NYSE holidays**
and **early closes**, Gateway/UI prompts, and host/network faults can still require human action.
Use supported unattended patterns (e.g. IBC ``AUTO_RESTART_TIME``, account settings) and alert on
failures (Telegram, journald). Equity orders from ``ibkr_equity_paper_trade.py`` are gated to **NYSE RTH**
by default; universe rescans use a separate **09:30–16:00 ET** scanner window (NYSE cash session).

Telegram
--------

Run **either** ``run_master.py --telegram-bot`` **or** a standalone ``rlm_telegram_bot`` service — **not both**.

Set ``Restart=always`` in the unit if you want restarts even after a clean exit
0 (at the cost of harder debugging if something exits "successfully" in a
loop).  The examples use ``always`` for long-running trading UI + pipeline
services; switch to ``on-failure`` if you prefer restarts only on errors.

IB Gateway in Docker: set ``--restart=unless-stopped`` (or Compose
``restart: unless-stopped``) on the *container*, not only the Java process, so
reboots and Docker daemon restarts bring the gateway back.

External monitoring (health checks, alerts when HTTP/TCP to your app fails) is
still recommended; systemd does not replace observability.
